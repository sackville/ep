# Conceptual Modeling Principles

## Purpose

The Conceptual Domain Model describes the truths that exist within Command.

It intentionally avoids implementation concerns such as persistence, APIs, services, user interface frameworks, or deployment architecture.

Its purpose is to establish the ubiquitous language shared by designers, engineers, founders, customers, and AI collaborators.

---

## Model Reality Before Implementation

The conceptual model describes reality.

Implementation describes one way to represent that reality.

Whenever implementation concerns influence the conceptual model, the model should be reconsidered.

---

## Distinguish Reality from Representation

Reality exists independently of the board.

The board represents the team's current shared understanding of reality.

As reality changes, the board should be updated to remain truthful.

---

## Discover Concepts

Concepts should be discovered through observation of real work rather than invented for convenience.

The persona, design principles, and design discussions should reveal the concepts that belong in the model.

---

## Relationships Carry Meaning

Nodes are important.

Relationships give them meaning.

The board exists not merely to display entities but to reveal how they influence one another.

Understanding emerges from relationships.

---

## The UI Reveals the Domain

The user interface should reveal concepts that already exist within the domain.

It should not invent concepts merely because they make visualization easier.

Whenever possible, visual elements should correspond directly to domain concepts.

---

## Views Are Not the Domain

Some concepts exist only as projections over the domain.

Examples include:

- Wake
- Frontier
- Seam

These are valuable visualizations, but they are not themselves business concepts.

The conceptual model should define the underlying truths from which these views are derived.

---

## Preserve a Ubiquitous Language

Every important concept should have one clear meaning.

The same terminology should be used consistently across:

- the Design Brief,
- the Conceptual Domain Model,
- documentation,
- implementation,
- conversations,
- and future products.

---

## Let the Domain Drive the Design

The visual language should emerge from the conceptual model.

Scene design, interaction design, and implementation should all reinforce the domain rather than redefine it.

When the domain is understood clearly, visualization becomes an act of revealing rather than inventing.
